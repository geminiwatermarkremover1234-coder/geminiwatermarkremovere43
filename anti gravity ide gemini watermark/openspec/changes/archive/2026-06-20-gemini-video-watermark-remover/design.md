## Context

The goal is to build a fully functional clone of the Gemini Video Watermark Remover. This web application runs completely client-side in the browser. It processes MP4 files frame-by-frame using the browser's WebCodecs API and canvas rendering to remove the visible watermark in the corner of video frames generated by Gemini (720p or 1080p).

## Goals / Non-Goals

**Goals:**
- Client-side-only processing: Decouple video editing from backend servers using WebCodecs (VideoDecoder/VideoEncoder) for speed, privacy, and cost efficiency.
- Dynamic watermark detection: Compute similarity/correlation scores to identify the exact corner offset of the watermark logo.
- Inverse-blend watermark erasure: Apply pixel unblending math to reverse the alpha channel blend and recover the original background video details.
- Audio track conservation: Demux and remux audio chunks directly from the source video into the output MP4 to keep audio in sync.
- Premium UI/UX: Create a dark mode interface with vanilla CSS variables, glassmorphic accents, responsive grid structures, file dropzones, queue tables, and credit alerts.

**Non-Goals:**
- Server-side video encoding/decoding.
- Removing watermark logos from arbitrary/uncalibrated resolutions or aspect ratios (outside of 720p/1080p landscape & portrait).
- Removing invisible watermarks, steganographic signatures, or metadata provenance.

## Decisions

### 1. Technology Stack
- **Choice**: Vanilla HTML5, modern ES6 JavaScript, and Vanilla CSS with custom properties (CSS variables).
- **Rationale**: Keeps the application lightweight, responsive, and easy to run locally or host on static providers (GitHub Pages, Netlify). Avoids Tailwind compilation overhead and fits strict instructions regarding styling.
- **Alternatives Considered**: React/Next.js (adds build-time complexity for a single-page app and increases resource footprints).

### 2. Video Processing Engine
- **Choice**: Browser WebCodecs API combined with `<canvas>` for image frame manipulation.
- **Rationale**: WebCodecs provides low-level access to the hardware-accelerated H.264 video decoder and encoder on Chromium-based browsers, making frame-by-frame processing fast enough to run in real-time.
- **Alternatives Considered**: FFmpeg.wasm (very heavy download ~30MB, slow execution speed, high memory consumption, and lack of browser-level optimization).

### 3. Demuxing and Muxing Libraries
- **Choice**: `mp4box.all.min.js` (for parsing MP4 atoms and samples) and `mp4-muxer.min.js` (for writing the final MP4 container).
- **Rationale**: Lightweight, robust, and execute locally in pure JS. They allow us to read/write H.264 streams and AAC audio packets seamlessly.
- **Alternatives Considered**: Writing custom container parsers (extremely bug-prone and complex).

### 4. Watermark Erasure Algorithm
- **Choice**: Math-based Inverse Alpha Blending + Bounding Box Detection.
  - Pearson correlation against template alpha masks (bg_48 for 720p, paired star logo for 1080p) to find coordinate list offset `x, y`.
  - Blending factor `p` extracted from template.
  - Background color formula `B = (Original - WatermarkColor * p) / (1 - p)` (assuming white watermark `WatermarkColor = 250`).
  - Edge dilution + local box average smoothing to hide compression blockiness.
- **Rationale**: Restores the actual video background with zero blur when the watermark is translucent. Blurs only the high-alpha edges.
- **Alternatives Considered**: Simple color-inpaint or Gaussian blur (looks blurry and unnatural, ruining video quality).

## Risks / Trade-offs

- **[Risk] Browser Compatibility** → WebCodecs is a modern API and might not be fully supported in Firefox or Safari.
  - *Mitigation*: Display a prominent warning message if WebCodecs APIs (`VideoDecoder` or `VideoEncoder`) are missing in the browser environment, prompting the user to use Chrome/Edge.
- **[Risk] Audio-Video Desync** → Frame rates might fluctuate, or audio codec descriptors might mismatch.
  - *Mitigation*: Extract audio sample timestamps (`cts`) and durations directly from MP4Box and copy them linearly with exact timestamps into the muxer.
